For a dTelecom client to successfully connect to the server, it must pass an access token with the request.

This token encodes the identity of a participant, name of the room, capabilities and permissions. Access tokens are JWT-based and signed with your API secret to prevent forgery.

**dTelecom uses Ed25519 signing** (not HMAC-SHA256 like standard LiveKit). Your API Key is a Solana Ed25519 public key, and your API Secret is the corresponding private key. You get these from [cloud.dtelecom.org](https://cloud.dtelecom.org) when you create a project.

Since participant identity is unique within the room, each access token can be used only by a single participant.

## Creating a token

```typescript title="TypeScript"
import { AccessToken } from '@dtelecom/server-sdk-js';

const roomName = 'name-of-room';
const participantIdentity = 'unique-identity';
const participantName = 'display name';

const at = new AccessToken('api-key', 'secret-key', {
  identity: participantIdentity,
  name: participantName,
});
at.addGrant({ roomJoin: true, room: roomName, canPublish: true, canSubscribe: true });

const token = at.toJwt();

// Get the WebSocket URL for the best SFU node based on client IP
const clientIp = '203.0.113.1'; // get from request headers (e.g. x-forwarded-for)
const wsUrl = await at.getWsUrl(clientIp);

console.log('access token', token);
console.log('websocket url', wsUrl);
// Return both token and wsUrl to the client
```

## Room permissions

Room permissions are specified in the `video` field of a decoded join token. It may contain one or more of the following properties:

| field        | type   | description                                               |
| :----------- | :----- | :-------------------------------------------------------- |
| roomCreate   | bool   | permission to create rooms                                |
| roomJoin     | bool   | permission to join a room                                 |
| roomAdmin    | bool   | permission to moderate a room                             |
| room         | string | name of the room, required if join or admin is set        |
| canPublish     | bool   | allow participant to publish tracks                       |
| canSubscribe   | bool   | allow participant to subscribe to tracks                  |
| canPublishData | bool   | allow participant to send data messages (needed for chat) |

### Example: subscribe-only token

To create a token where the participant can only subscribe, and not publish into the room, you would use the following grant:

```json
{
  ...
  "video": {
    "room": "myroom",
    "roomJoin": true,
    "canSubscribe": true,
    "canPublish": false,
    "canPublishData": false
  }
}
```

## Webhook URL

You can embed a webhook URL in the token so the SFU node sends room events (participant joined/left, room finished) to your server. See [Webhooks](/guides/webhooks) for details.

```typescript
const at = new AccessToken(apiKey, apiSecret, {
  identity: 'user-1',
  name: 'Alice',
  webHookURL: process.env.WEBHOOK_URL, // e.g. https://my-app.example.com/api/webhook
});
```

## Token expiration

A tokens has an expiration time. We recommend setting it to the expected duration of a session. A dTelecom client will store a token for the entire session duration, in case it needs to reconnect.

## Participant metadata

You may also attach any arbitrary metadata onto each participant via the `metadata` field. This data is opaque to dTelecom.

When provided, dTelecom will attach the metadata to the participant object that the clients would receive.

### Role-based metadata example

A common pattern is embedding a role (e.g. `host` or `guest`) in the participant metadata. This lets the client show different UI based on role, and the backend can verify the role before allowing privileged actions.

**Server-side** — set role when creating the token:

```typescript
const metadata = JSON.stringify({ role: 'host' }); // or 'guest'

const at = new AccessToken(apiKey, apiSecret, {
  identity: 'user-1',
  name: 'Alice',
  metadata,
});
at.addGrant({ roomJoin: true, room: 'my-room', canPublish: true, canSubscribe: true, canPublishData: true });
```

**Client-side** — read role from metadata:

```tsx
import { useLocalParticipant } from '@dtelecom/components-react';

function HostBadge() {
  const { localParticipant } = useLocalParticipant();
  const meta = JSON.parse(localParticipant.metadata || '{}');

  if (meta.role === 'host') {
    return <span>Host</span>;
  }
  return null;
}
```

:::caution Security note
Always verify the participant's role **server-side** before executing privileged actions (kick, mute, etc.). The client-side metadata check is only for UI rendering — a malicious client could modify its own UI. See [Host controls pattern](/guides/server-api#host-controls-pattern) for the full flow.
:::
